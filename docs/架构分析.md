# Elpis-core 架构分析文档

## 一、架构定性：CSR（客户端渲染）多页应用（MPA）

Elpis-core 是一个典型的 **CSR（Client-Side Rendering）** 架构项目，采用 **Koa 服务端 + Vue 3 客户端** 的前后端一体化模式，同时结合了**多页应用（MPA）**的打包策略。

> **一句话总结**：服务端（Koa）负责路由分发、API 服务和返回 HTML 空壳，客户端（Vue 3）负责页面渲染和交互逻辑。Nunjucks 模板引擎仅作为二者之间的"胶水层"。

---

## 二、技术栈总览

| 层级 | 技术选型 | 职责 |
|------|---------|------|
| **服务端框架** | Koa 2 | Web 服务、路由、中间件 |
| **模板引擎** | koa-nunjucks-2 | 渲染 HTML 壳页面（非业务 HTML） |
| **前端框架** | Vue 3 | 页面组件化渲染、交互 |
| **UI 组件库** | Element Plus | 表单、表格、弹窗等 UI 组件 |
| **状态管理** | Pinia | 全局状态管理 |
| **前端路由** | Vue Router 4（Hash 模式） | 页面内子路由 |
| **构建工具** | Webpack 5 | 多入口打包 |
| **数据库** | MySQL（via knex） | 数据持久化 |
| **HTTP 客户端** | axios / superagent | 前后端 HTTP 请求 |

```
┌────────────────────────────────────────────────────────────────┐

│                                                                │
│   1. 用户访问 /view/dashboard                                   │
│           ↓                                                    │
│   2. Koa 返回 entry.dashboard.tpl（HTML 空壳）                   │

│           ↓                                                    │
│   5. vue-router（Hash 模式）管理页面内子路由                      │
```

## 四、CSR 架构的关键证据

### 4.1 服务端只返回"空壳 HTML"

`app/view/entry.tpl`：

```html
<!DOCTYPE html>
<html class="dark">
<head>
  <meta charset="utf-8">
  <link href="/static/normalize.css" rel="stylesheet">
  <link href="/static/logo.png" rel="icon" type="image/x-icon">
</head>
<body style="margin:0">
 <div id="root"></div>
 <input id="env" value="{{ env }}" style="display: none;">
 <input id="options" value="{{ options }}" style="display: none;">
</body>
<script type="text/javascript">
  try{
    window.env=document.getElementById('env').value;
    const options =document.getElementById('options').value;
    window.options=JSON.parse(options);
  }
  catch(e){ console.log(e); }
</script>
</html>
```

- `<div id="root"></div>` 是空容器，**没有任何服务端渲染的 Vue 组件内容**
- Nunjucks 仅注入 `env` 和 `options` 配置变量
- Webpack 构建时通过 `html-webpack-plugin` 将 JS bundle 注入到此模板中

### 4.2 Vue 在浏览器端创建并挂载应用

`app/pages/boot.js`：

```javascript
import { createApp } from "vue";                    // ← 非 createSSRApp
import { createRouter, createWebHashHistory } from "vue-router";

export default (pageComponent, { routers = [], libs = [] } = {}) => {
    const app = createApp(pageComponent);         // ← 客户端创建 Vue 实例
    app.use(ElementUi);
    app.use(pinia);

    const router = createRouter({
        history: createWebHashHistory(),           // ← Hash 路由，纯客户端
        routes: routers,
    });
    app.use(router);
    router.isReady().then(() => {
        app.mount('#root');                            // ← 客户端挂载
    });
}
```

**关键特征**：
- 使用 `createApp()` 而非 SSR 的 `createSSRApp()`
- 使用 `createWebHashHistory()` 而非 SSR 常用的 `createMemoryHistory()`
- 使用 `app.mount('#root')` 客户端挂载，而非 SSR 的 `renderToString()`

### 4.3 没有任何 SSR 相关依赖

在 `package.json` 中：
- ❌ 没有 `vue-server-renderer`
- ❌ 没有 `@vue/server-renderer`
- ❌ 没有 `vue-bundle-renderer`
- ❌ Webpack 配置中没有 `target: 'node'`

## 五、为什么选择此架构？—— 融合方案的设计动机

### 5.1 两个极端方案的困境

#### 极端一：纯 Nunjucks 模板渲染（传统 SSR）

```html
<!-- Nunjucks 模板中处理复杂交互 -->
<table>
  {% for item in list %}
    <tr>
      <td>{{ item.name }}</td>
      <td><button onclick="handleEdit({{ item.id }})">编辑</button></td>
    </tr>
  {% endfor %}
</table>
<script>
function handleEdit(id) {
  // 弹出弹窗 → 手动创建 DOM
  // 表单验证 → 手动逐字段检查
  // 提交后刷新 → 手动操作 DOM 或刷新整页
}
</script>
```

| 优势 | 劣势 |
|------|------|
| ✅ SEO 友好 | ❌ 复杂交互需大量手写 DOM 操作 |
| ✅ 首屏直出 | ❌ 无法组件化复用 |
| | ❌ 无法使用 Element Plus 等 UI 库 |
| | ❌ 状态管理混乱，数据散落在 DOM 中 |
| | ❌ 开发效率低，维护成本高 |

#### 极端二：纯 Vue SPA（完全 CSR）

```javascript
// 所有路由在前端定义，服务端退化为纯 API
const router = createRouter({
  routes: [
    { path: '/dashboard', component: Dashboard },
    { path: '/project-list', component: ProjectList },
    { path: '/user-center', component: UserCenter },
    // ... 几十个页面
  ]
})
```

| 优势 | 劣势 |
|------|------|
| ✅ 组件化开发 | ❌ 服务端失去路由控制权 |
| ✅ 复杂交互友好 | ❌ 无法在服务端做页面级权限拦截 |
| ✅ 可用 UI 组件库 | ❌ 所有页面打包成一个巨大 JS，首屏慢 |
| | ❌ SEO 不友好 |

### 5.2 为什么纯 SPA 会导致"巨大 JS 包"？

在标准 SPA 中，只有**一个入口文件**，Webpack 从这个入口沿 `import` 链追踪所有依赖：

```
main.js (唯一入口)
 ├── App.vue
 ├── router/index.js
 │    ├── Dashboard.vue → echarts (800KB)
 │    ├── ProjectList.vue → element-plus/table
 │    ├── UserCenter.vue
 │    ├── Settings.vue
 │    └── Reports.vue → xlsx-parser (500KB)
 ├── element-plus (全量引入)
 ├── pinia
 └── axios
```

打包结果：

```
dist/
  index.html           ← 空壳
  js/app.3a8f2c.js     ← 2~5MB（所有页面 + 所有依赖）
  js/vendor.1b2c3d.js  ← 第三方库
```

**首屏加载流程**：

```
用户访问 /dashboard
    ↓
下载 app.js (3MB) + vendor.js (2MB)  ← 即使只看 Dashboard 也要下载全部
    ↓
浏览器解析、编译、执行 JS
    ↓
Vue 渲染页面
    ↓
用户看到内容（3~5 秒后）
```

虽然 Vue Router 支持路由懒加载（`component: () => import('./xxx.vue')`），但只能缓解而非根治：
- 核心包（Vue + Router + 全局 Element Plus）仍然很大
- 第三方库 `vendor.js` 仍全量打包
- 路由表、导航守卫、全局插件仍在首屏加载

### 5.3 Elpis-core 的融合方案

Elpis-core 的设计哲学：**让 Koa 做它擅长的事（路由分发、中间件、API），让 Vue 做它擅长的事（组件化、交互、状态管理），Nunjucks 仅作为"胶水层"。**

## 六、MPA 多入口打包策略

### 6.1 Webpack 多入口配置

`app/webpack/config/webpack.base.js`：

```javascript
const pageEntries = {};
const htmlWebpackPluginList = [];

// 扫描 app/pages 下所有 entry.*.js 文件
const entryList = path.resolve(process.cwd(), './app/pages/**/entry.*.js');
glob.sync(entryList).forEach(file => {
  const entryName = path.basename(file, '.js');
  // 每个 entry.*.js 作为独立入口
  pageEntries[`entry.${entryName}`] = file;
  // 每个入口生成独立的 .tpl 模板
  htmlWebpackPluginList.push(
    new HtmlWebpackPlugin({
      filename: path.resolve(process.cwd(), './app/public/dist/', `${entryName}.tpl`),
      template: path.resolve(process.cwd(), './app/view/entry.tpl'),
      chunks: [`entry.${entryName}`],   // ← 只注入当前页面的 chunk
    })
  );
});
```

### 6.2 打包产物对比

```
【SPA 单入口】                          【MPA 多入口（Elpis）】
dist/                                   dist/
  index.html                              entry.dashboard.tpl
  js/                                     entry.project-list.tpl
    app.js        ← 3MB 全部页面           js/
    vendor.js     ← 2MB 全部依赖             entry.entry.dashboard.js    ← 200KB
                                             entry.entry.project-list.js ← 150KB
                                             vendors.js                  ← 800KB (共享)
                                             common.js                   ← 100KB (共享)
```

### 6.3 代码分割策略

```javascript
optimization: {
  splitChunks: {
    chunks: 'all',
    cacheGroups: {
      vendors: {
        test: /[\\/]node_modules[\\/]/,
        name: 'vendors',          // 第三方库提取为公共 chunk
        priority: 20,
      },
      commons: {
        test: /[\\/]common|widgets[\\/]/,
        name: 'common',           // 公共业务组件提取为公共 chunk
        minChunks: 2,
        priority: 10,
      },
    },
  },
}
```

将 JS 打包为三层：
1. **vendors**：第三方库（Vue、Element Plus 等），基本不改动
2. **commons**：公共业务组件（common、widgets 目录），改动较少
3. **entry.{page}**：各页面独有代码，经常改动

> 目的：按改动频率分层，最大化利用浏览器缓存。

## 七、请求流程详解

### 7.1 页面请求流程

```
1. 用户访问 http://localhost:8080/view/dashboard
        ↓
2. Koa 路由匹配 /view/:page  (app/router/view.js)
        ↓
3. 中间件链执行：
   - koa-static（静态资源）
   - koa-nunjucks-2（模板引擎）
   - koa-bodyparser（请求体解析）
   - api-sign-verify（签名校验，仅 /api 路径）
   - error-handle（异常捕获）
   - api-params-verify（参数校验，仅 /api 路径）
        ↓
4. ViewController.renderPage 执行
   ctx.render('dist/entry.dashboard', { env, options })
        ↓
5. Nunjucks 渲染 entry.dashboard.tpl → 返回 HTML
   （HTML 中已注入 entry.dashboard.js 的 <script> 标签）
        ↓
6. 浏览器接收 HTML → 加载 JS bundle
        ↓
7. Vue 3 createApp() → mount('#root') → 页面渲染完成
        ↓
8. 页面内通过 vue-router (Hash模式) 管理子路由
   如 #/schema、#/iframe、#/todo
```

### 7.2 API 请求流程

```
1. 前端 axios 请求 /api/project/list
        ↓
2. Koa 路由匹配 (app/router/project.js)
        ↓
3. 中间件链执行：
   - api-sign-verify（验证签名）
   - api-params-verify（JSON Schema 参数校验）
        ↓
4. projectController.getList 执行
        ↓
5. projectService 调用数据库/模型获取数据
        ↓
6. 返回 JSON: { success: true, data: [...] }
```

## 八、方案对比总结

| 维度 | 纯 Nunjucks SSR | 纯 Vue SPA | Elpis MPA 融合方案 |
|------|-----------------|-----------|-------------------|
| **服务端路由控制** | ✅ | ❌ | ✅ |
| **服务端中间件拦截** | ✅ | ❌ | ✅ |
| **组件化开发** | ❌ | ✅ | ✅ |
| **复杂交互处理** | ❌ | ✅ | ✅ |
| **UI 组件库支持** | ❌ | ✅ | ✅ |
| **状态管理** | ❌ | ✅ | ✅ |
| **按页面拆分 JS** | ✅（无 JS 包） | ❌ | ✅ |
| **首屏加载速度** | 快（直出 HTML） | 慢（大 JS 包） | 较快（小 JS 包） |
| **SEO 友好** | ✅ | ❌ | ⚠️ 部分 |
| **页面间切换** | 整页刷新 | 无刷新 | 整页刷新 |
| **开发效率** | 低 | 高 | 高 |
| **适用场景** | 内容展示站 | C端应用 | **后台管理系统** |

## 九、结论

Elpis-core 通过 **MPA + CSR** 的融合架构，在 Koa 的后端控制力和 Vue 的前端表现力之间找到了平衡点：

- **Koa**：掌控路由入口、中间件（鉴权/校验/异常处理）、API 服务
- **Nunjucks**：仅作为"胶水层"，将服务端配置注入到 HTML 壳中
- **Vue 3 + Element Plus**：接管所有页面渲染和用户交互
- **Webpack 多入口**：每个页面独立打包，避免 SPA 的大包问题

这种架构特别适合**后台管理系统**类场景：页面间跳转频率低（整页刷新可接受），页面内交互复杂（需要 Vue 组件化能力），同时需要服务端保持路由和权限控制。
