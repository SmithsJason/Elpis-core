# Elpis-core 架构分析文档

## 一、架构定性：CSR（客户端渲染）多页应用（MPA）

Elpis-core 是一个典型的 **CSR（Client-Side Rendering）** 架构项目，采用 **Koa 服务端 + Vue 3 客户端** 的前后端一体化模式，同时结合了 **多页应用（MPA）** 的打包策略。

> **一句话总结**：服务端（Koa）负责路由分发、API 服务和返回 HTML 空壳，客户端（Vue 3）负责页面渲染和交互逻辑。Nunjucks 模板引擎仅作为二者之间的"胶水层"。

---

## 二、技术栈总览

| 层级 | 技术选型 | 职责 |
|------|---------|------|
| **服务端框架** | Koa 2 | Web 服务、路由、中间件 |
| **模板引擎** | koa-nunjucks-2 | 渲染 HTML 壳页面（非业务 HTML） |
| **前端框架** | Vue 3 | 页面组件化渲染、交互 |
| **UI 组件库** | Element Plus | 表单、表格、弹窗等 UI 组件 |
| **状态管理** | Pinia | 全局状态管理 |
| **前端路由** | Vue Router 4（Hash 模式） | 页面内子路由 |
| **构建工具** | Webpack 5 | 多入口打包 |
| **数据库** | MySQL（via knex） | 数据持久化 |
| **HTTP 客户端** | axios / superagent | 前后端 HTTP 请求 |

│                                                                │
│   1. 用户访问 /view/dashboard                                   │

│   /             → 302 重定向到 homePage                          │

## 四、CSR 架构的关键证据

### 4.1 服务端只返回"空壳 HTML"

`app/view/entry.tpl`：

```html
<!DOCTYPE html>
<html class="dark">
<head>
  <meta charset="utf-8">
  <link href="/static/normalize.css" rel="stylesheet">
  <link href="/static/logo.png" rel="icon" type="image/x-icon">
</head>
<body style="margin:0">
 <div id="root"></div>
 <input id="env" value="{{ env }}" style="display: none;">
 <input id="options" value="{{ options }}" style="display: none;">
</body>
<script type="text/javascript">
  try{
    window.env=document.getElementById('env').value;
    const options =document.getElementById('options').value;
    window.options=JSON.parse(options);
  }
  catch(e){ console.log(e); }
</script>
</html>
```

- `<div id="root"></div>` 是空容器，**没有任何服务端渲染的 Vue 组件内容**
- Nunjucks 仅注入 `env` 和 `options` 配置变量
- Webpack 构建时通过 `html-webpack-plugin` 将 JS bundle 注入到此模板中

### 4.2 Vue 在浏览器端创建并挂载应用

`app/pages/boot.js`：

```javascript
import { createApp } from "vue";                    // ← 非 createSSRApp
import { createRouter, createWebHashHistory } from "vue-router";

export default (pageComponent, { routers = [], libs = [] } = {}) => {
    const app = createApp(pageComponent);         // ← 客户端创建 Vue 实例
    app.use(ElementUi);
    app.use(pinia);

    const router = createRouter({
        history: createWebHashHistory(),           // ← Hash 路由，纯客户端
        routes: routers,
    });
    app.use(router);
    router.isReady().then(() => {
        app.mount('#root');                            // ← 客户端挂载
    });
}
```

**关键特征**：
- 使用 `createApp()` 而非 SSR 的 `createSSRApp()`
- 使用 `createWebHashHistory()` 而非 SSR 常用的 `createMemoryHistory()`
- 使用 `app.mount('#root')` 客户端挂载，而非 SSR 的 `renderToString()`

### 4.3 没有任何 SSR 相关依赖

在 `package.json` 中：
- ❌ 没有 `vue-server-renderer`
- ❌ 没有 `@vue/server-renderer`
- ❌ 没有 `vue-bundle-renderer`
- ❌ Webpack 配置中没有 `target: 'node'`

## 五、为什么选择此架构？—— 融合方案的设计动机

### 5.1 两个极端方案的困境

#### 极端一：纯 Nunjucks 模板渲染（传统 SSR）

```html
<!-- Nunjucks 模板中处理复杂交互 -->
<table>
  {% for item in list %}
    <tr>
      <td>{{ item.name }}</td>
      <td><button onclick="handleEdit({{ item.id }})">编辑</button></td>
    </tr>
  {% endfor %}
</table>
<script>
function handleEdit(id) {
  // 弹出弹窗 → 手动创建 DOM
  // 表单验证 → 手动逐字段检查
  // 提交后刷新 → 手动操作 DOM 或刷新整页
}
</script>
```

| 优势 | 劣势 |
|------|------|
| ✅ SEO 友好 | ❌ 复杂交互需大量手写 DOM 操作 |
| ✅ 首屏直出 | ❌ 无法组件化复用 |
| | ❌ 无法使用 Element Plus 等 UI 库 |
| | ❌ 状态管理混乱，数据散落在 DOM 中 |
| | ❌ 开发效率低，维护成本高 |

#### 极端二：纯 Vue SPA（完全 CSR）

```javascript
// 所有路由在前端定义，服务端退化为纯 API
const router = createRouter({
  routes: [
    { path: '/dashboard', component: Dashboard },
    { path: '/project-list', component: ProjectList },
    { path: '/user-center', component: UserCenter },
    // ... 几十个页面
  ]
})
```

| 优势 | 劣势 |
|------|------|
| ✅ 组件化开发 | ❌ 服务端失去路由控制权 |
| ✅ 复杂交互友好 | ❌ 无法在服务端做页面级权限拦截 |
| ✅ 可用 UI 组件库 | ❌ 所有页面打包成一个巨大 JS，首屏慢 |
| | ❌ SEO 不友好 |

### 5.2 为什么纯 SPA 会导致"巨大 JS 包"？

在标准 SPA 中，只有**一个入口文件**，Webpack 从这个入口沿 `import` 链追踪所有依赖：

```
main.js (唯一入口)
 ├── App.vue
 ├── router/index.js
 │    ├── Dashboard.vue → echarts (800KB)
 │    ├── ProjectList.vue → element-plus/table
 │    ├── UserCenter.vue
 │    ├── Settings.vue
 │    └── Reports.vue → xlsx-parser (500KB)
 ├── element-plus (全量引入)
 ├── pinia
 └── axios
```

打包结果：

```
dist/
  index.html           ← 空壳
  js/app.3a8f2c.js     ← 2~5MB（所有页面 + 所有依赖）
  js/vendor.1b2c3d.js  ← 第三方库
```

**首屏加载流程**：

```
用户访问 /dashboard
    ↓
下载 app.js (3MB) + vendor.js (2MB)  ← 即使只看 Dashboard 也要下载全部
    ↓
浏览器解析、编译、执行 JS
    ↓
Vue 渲染页面
    ↓
用户看到内容（3~5 秒后）
```

虽然 Vue Router 支持路由懒加载（`component: () => import('./xxx.vue')`），但只能缓解而非根治：
- 核心包（Vue + Router + 全局 Element Plus）仍然很大
- 第三方库 `vendor.js` 仍全量打包
- 路由表、导航守卫、全局插件仍在首屏加载

### 5.3 Elpis-core 的融合方案

Elpis-core 的设计哲学：**让 Koa 做它擅长的事（路由分发、中间件、API），让 Vue 做它擅长的事（组件化、交互、状态管理），Nunjucks 仅作为"胶水层"。**

## 六、MPA 多入口打包策略

### 6.1 Webpack 多入口配置

`app/webpack/config/webpack.base.js`：

```javascript
const pageEntries = {};
const htmlWebpackPluginList = [];

// 扫描 app/pages 下所有 entry.*.js 文件
const entryList = path.resolve(process.cwd(), './app/pages/**/entry.*.js');
glob.sync(entryList).forEach(file => {
  const entryName = path.basename(file, '.js');
  // 每个 entry.*.js 作为独立入口
  pageEntries[`entry.${entryName}`] = file;
  // 每个入口生成独立的 .tpl 模板
  htmlWebpackPluginList.push(
    new HtmlWebpackPlugin({
      filename: path.resolve(process.cwd(), './app/public/dist/', `${entryName}.tpl`),
      template: path.resolve(process.cwd(), './app/view/entry.tpl'),
      chunks: [`entry.${entryName}`],   // ← 只注入当前页面的 chunk
    })
  );
});
```

### 6.2 打包产物对比

```
【SPA 单入口】                          【MPA 多入口（Elpis）】
dist/                                   dist/
  index.html                              entry.dashboard.tpl
  js/                                     entry.project-list.tpl
    app.js        ← 3MB 全部页面           js/
    vendor.js     ← 2MB 全部依赖             entry.entry.dashboard.js    ← 200KB
                                             entry.entry.project-list.js ← 150KB
                                             vendors.js                  ← 800KB (共享)
                                             common.js                   ← 100KB (共享)
```

### 6.3 代码分割策略

```javascript
optimization: {
  splitChunks: {
    chunks: 'all',
    cacheGroups: {
      vendors: {
        test: /[\\/]node_modules[\\/]/,
        name: 'vendors',          // 第三方库提取为公共 chunk
        priority: 20,
      },
      commons: {
        test: /[\\/]common|widgets[\\/]/,
        name: 'common',           // 公共业务组件提取为公共 chunk
        minChunks: 2,
        priority: 10,
      },
    },
  },
}
```

将 JS 打包为三层：
1. **vendors**：第三方库（Vue、Element Plus 等），基本不改动
2. **commons**：公共业务组件（common、widgets 目录），改动较少
3. **entry.{page}**：各页面独有代码，经常改动

> 目的：按改动频率分层，最大化利用浏览器缓存。

## 七、请求流程详解

### 7.1 页面请求流程

```
1. 用户访问 http://localhost:8080/view/dashboard
        ↓
2. Koa 路由匹配 /view/:page  (app/router/view.js)
        ↓
3. 中间件链执行：
   - koa-static（静态资源）
   - koa-nunjucks-2（模板引擎）
   - koa-bodyparser（请求体解析）
   - api-sign-verify（签名校验，仅 /api 路径）
   - error-handle（异常捕获）
   - api-params-verify（参数校验，仅 /api 路径）
        ↓
4. ViewController.renderPage 执行
   ctx.render('dist/entry.dashboard', { env, options })
        ↓
5. Nunjucks 渲染 entry.dashboard.tpl → 返回 HTML
   （HTML 中已注入 entry.dashboard.js 的 <script> 标签）
        ↓
6. 浏览器接收 HTML → 加载 JS bundle
        ↓
7. Vue 3 createApp() → mount('#root') → 页面渲染完成
        ↓
8. 页面内通过 vue-router (Hash模式) 管理子路由
   如 #/schema、#/iframe、#/todo
```

### 7.2 API 请求流程

```
1. 前端 axios 请求 /api/project/list
        ↓
2. Koa 路由匹配 (app/router/project.js)
        ↓
3. 中间件链执行：
   - api-sign-verify（验证签名）
   - api-params-verify（JSON Schema 参数校验）
        ↓
4. projectController.getList 执行
        ↓
5. projectService 调用数据库/模型获取数据
        ↓
6. 返回 JSON: { success: true, data: [...] }
```

## 九、"服务端提供模板，客户端注入"的设计动机

### 9.1 核心问题：打破启动配置的循环依赖

纯 CSR 有一个先天缺陷：**客户端在 JS 加载并执行之前，对服务端的运行时信息一无所知**。

Elpis-core 通过服务端模板注入来解决这个问题。`app/controller/view.js` 的核心逻辑：

```javascript
async renderPage(ctx) {
    await ctx.render(`dist/entry.${ctx.params.page}`, {
        name: app.options.name,
        env: app.env.getEnv(),                    // ← 注入当前运行环境（local/beta/prod）
        options: JSON.stringify(app.options),      // ← 注入整个应用配置对象
    });
}
```

`app/view/entry.tpl` 中，这些变量通过隐藏 `<input>` 被"种"到页面里：

```html
<input id="env" value="{{ env }}" style="display: none;">
<input id="options" value="{{ options }}" style="display: none;">
```

客户端 JS 一启动就能立即获取：

```javascript
window.env = document.getElementById('env').value;
window.options = JSON.parse(document.getElementById('options').value);
```

**为什么不用 API 请求来获取这些配置？** 因为这些是"启动前置依赖"——Vue 应用还没创建（`createApp` 还没执行），路由还没初始化，axios 还没配置好。如果用 API 请求，会出现**循环依赖**：

```
要发 API 请求 → 需要知道当前环境的 API 地址
要知道 API 地址 → 需要先获取 env 配置
要获取配置     → 需要发 API 请求...
```

服务端模板注入打破了这个循环——**在 HTML 返回的那一刻，配置就已经在页面里了**。

### 9.2 多环境部署：一份构建产物，多环境运行

Elpis-core 的 `options` 包含整个应用的运行时配置，这些配置是**环境相关**的：

| 环境 | `env` 值 | 配置差异 |
|------|---------|---------|
| 本地开发 | `local` | 连本地 MySQL、开启 debug |
| 测试环境 | `beta` | 连测试库、API 地址不同 |
| 生产环境 | `production` | 连生产库、关闭 debug |

如果硬编码在前端 JS 里：
- 每次切换环境都要**重新构建前端**
- 无法做到**一份构建产物，多环境部署**
- 配置泄露风险更高

通过服务端注入，**同一份 Webpack 构建产物可以在任何环境运行**，因为配置是运行时由 Koa 动态注入的。

### 9.3 实际落地业务场景

从项目的 `model` 配置可以看到，Elpis-core 是一个**多租户/多项目管理后台框架**，典型业务场景如下：

#### 场景一：多电商平台统一管理后台

```
model/business/ (电商系统模型)
├── model.js          → 定义通用菜单：商品管理、订单管理、客户管理
└── project/
    ├── taobao.js     → 淘宝：覆盖订单管理为 iframe 嵌入、新增运营活动模块
    ├── pdd.js        → 拼多多：覆盖客户管理为 schema 表单、新增数据分析模块
    └── jd.js         → 京东：新增店铺设置（分组菜单）
```

**场景描述**：一家公司同时在淘宝、拼多多、京东开店，需要一个统一的后台来管理所有店铺。每个平台的管理需求有共性（都有商品、订单、客户），但也有差异（拼多多有拼团数据分析，京东有店铺资质管理）。

#### 场景二：多平台在线课程管理

```
model/course/ (课程系统模型)
├── model.js          → 定义通用菜单：视频管理、客户管理
└── project/
    ├── bilibili.js   → B站课堂：重命名菜单、新增课程资料模块
    └── douyin.js     → 抖音课堂：新增流量管理模块
```

**场景描述**：一家教育公司在 B 站和抖音同时开设课程，需要一个后台统一管理。B 站侧重课程资料分发（PDF/Excel/PPT），抖音侧重流量运营。

#### 场景三：更通用的落地场景

"**模型定义通用能力 → 项目继承并差异化**"的模式，在企业级应用中非常常见：

| 领域 | model 层（通用） | project 层（差异化） |
|------|----------------|-------------------|
| **SaaS 运营后台** | 用户管理、权限管理、数据看板 | 不同客户定制化的功能模块 |
| **多品牌管理** | 商品管理、库存管理、供应链 | 各品牌独有的营销策略配置 |
| **多区域运营** | 通用运营工具、报表系统 | 各地区特有的合规/支付/物流配置 |
| **低代码平台** | Schema 驱动的表单/表格 | 不同业务方的自定义字段和流程 |

## 十、服务端路由控制 vs SPA 前端路由控制

### 10.1 纯 SPA 的路由控制 —— 浏览器端拦截

```
用户输入 http://xxx.com/dashboard
        ↓
服务端：无论你请求什么路径，统一返回同一个 index.html
        ↓
浏览器加载 JS（可能 3MB）
        ↓
Vue Router 在浏览器里匹配路由 → 渲染 Dashboard 组件
        ↓
Vue Router 的 beforeEach 守卫做权限判断
        ↓
没权限？→ router.push('/login')  （仅仅是前端跳转）
```

**存在的问题**：

1. **JS 已经全部下载到浏览器了**。即使用户没有权限访问 `/admin`，admin 页面的代码（组件、业务逻辑）已经在用户手里了——用户打开 DevTools 就能看到。

2. **前端路由守卫可以被绕过**。用户只要在浏览器控制台执行：
   ```javascript
   localStorage.setItem('role', 'admin');
   router.push('/admin');
   ```
   前端守卫就形同虚设，因为**判断逻辑和数据都在客户端**。

3. **服务端完全不知道用户在访问什么页面**。对服务端来说，无论用户访问 `/dashboard` 还是 `/admin`，看到的请求都只是"给我 `index.html`"。

### 10.2 Elpis-core 的路由控制 —— 服务端拦截

```
用户输入 http://xxx.com/view/dashboard
        ↓
请求到达 Koa 服务端
        ↓
Koa 中间件链依次执行：
  ① error-handle（异常兜底）
  ② api-sign-verify（签名校验）
  ③ 可以在这里加：权限校验中间件 ← 【关键！请求还没到 Controller】
        ↓
匹配路由 /view/:page → ViewController.renderPage
        ↓
服务端决定：返回 entry.dashboard.tpl（只包含 dashboard 的 JS）
        ↓
浏览器只拿到 dashboard 页面的代码，根本没有 admin 页面的代码
```

`/view/:page` 是一个**服务端路由**（`app/router/view.js`），每次用户访问不同页面，都是一个**全新的 HTTP 请求**打到服务端。

服务端在返回 HTML 之前，可以做任何事情：

```javascript
router.get('/view/:page', 
    authMiddleware,             // ← 鉴权：没登录？直接 302 到登录页，JS 一行都不返回
    roleMiddleware,             // ← 权限：不是管理员？直接 403，admin 页面代码不给你
    ViewController.renderPage   // ← 通过了才返回页面
);
```

### 10.3 核心差异对比

```
【纯 SPA】
┌──────────────────────────────────────────────┐
│  服务端                                       │
│  GET /任何路径  →  统一返回 index.html          │
│  （服务端是"瞎子"，不知道用户在看哪个页面）       │
└──────────────────────────────────────────────┘
        ↓ 返回 index.html + 3MB JS（包含所有页面）
┌──────────────────────────────────────────────┐
│  浏览器                                       │
│  Vue Router 匹配 → beforeEach 守卫 → 渲染      │
│  ⚠️ 所有页面代码都已在浏览器里                   │
│  ⚠️ 权限判断在客户端，可被篡改                   │
└──────────────────────────────────────────────┘

【Elpis MPA】
┌──────────────────────────────────────────────┐
│  服务端                                       │
│  GET /view/dashboard → 中间件链 → 返回 dashboard │
│  GET /view/admin     → 中间件链 → 403 拒绝      │
│  ✅ 服务端清楚地知道用户请求的是哪个页面           │
│  ✅ 在返回任何 JS 之前就能拦截                    │
└──────────────────────────────────────────────┘
        ↓ 仅返回 dashboard 的 200KB JS
┌──────────────────────────────────────────────┐
│  浏览器                                       │
│  Vue 渲染 dashboard 页面                       │
│  ✅ 浏览器里根本没有 admin 页面的代码             │
└──────────────────────────────────────────────┘
```

### 10.4 具体权限控制示例

假设后台有 3 种角色：

| 角色 | 可访问页面 |
|------|-----------|
| 运营 | dashboard、数据分析 |
| 管理员 | dashboard、数据分析、用户管理 |
| 超级管理员 | 全部页面 |

**纯 SPA 的做法**：

```javascript
// 前端路由守卫
router.beforeEach((to) => {
  const role = localStorage.getItem('role'); // ← 从本地存储读角色
  if (to.path === '/user-manage' && role !== 'admin') {
    return '/403';
  }
});
```

问题：用户 `localStorage.setItem('role', 'admin')` 就绕过了。`user-manage` 组件的 JS 代码已经下载到浏览器了，至多只能"不显示页面"，但**代码和业务逻辑已经泄露**。

**Elpis MPA 的做法**：

```javascript
// 服务端中间件
async function roleCheck(ctx, next) {
  const user = await getUserFromSession(ctx);  // 从服务端 session 获取
  const page = ctx.params.page;
  
  if (page === 'user-manage' && user.role !== 'admin') {
    ctx.status = 403;
    return;  // ← 直接拒绝，一个字节的页面代码都不返回
  }
  await next();
}
```

用户在浏览器端**完全无法绕过**，因为判断逻辑在服务端，session 数据也在服务端。

### 10.5 路由控制安全级别总结

| 维度 | SPA 前端路由守卫 | Elpis 服务端中间件 |
|------|-----------------|-------------------|
| **控制时机** | JS 加载完毕后 | HTTP 请求到达时 |
| **判断依据** | 客户端存储（可篡改） | 服务端 session/DB（不可篡改） |
| **代码泄露** | 所有页面代码都在浏览器 | 未授权页面的代码不会下发 |
| **可绕过性** | 高（修改 localStorage/cookie） | 低（需要攻破服务端） |
| **适用场景** | C 端应用（安全要求低） | **管理后台（安全要求高）** |

对于 Elpis-core 的目标场景——**后台管理系统**，有些页面可能涉及用户数据、订单数据、财务数据，"代码不应该泄露给无权限的人"是一个硬性要求。SPA 配合后端 API 鉴权只能保护"数据"，而页面的结构、字段名、业务逻辑仍然暴露在前端代码中。Elpis 的方案是：**连代码都不给你。**

## 十二、结论

Elpis-core 通过 **MPA + CSR** 的融合架构，在 Koa 的后端控制力和 Vue 的前端表现力之间找到了平衡点：

- **Koa**：掌控路由入口、中间件（鉴权/校验/异常处理）、API 服务
- **Nunjucks**：仅作为"胶水层"，将服务端配置注入到 HTML 壳中，解决客户端启动配置的循环依赖问题
- **Vue 3 + Element Plus**：接管所有页面渲染和用户交互
- **Webpack 多入口**：每个页面独立打包，避免 SPA 的大包问题
- **服务端路由控制**：在 HTTP 层面实现页面级权限隔离，未授权页面的代码不会下发到浏览器

这种架构不是为了做 SSR，而是一种**务实的工程折中**——服务端保留了路由控制权、中间件鉴权能力和运行时配置注入能力，客户端获得了 Vue 组件化、Element Plus UI 库和 Pinia 状态管理等现代前端能力。特别适合**后台管理系统**类场景：页面间跳转频率低（整页刷新可接受），页面内交互复杂（需要 Vue 组件化能力），同时需要服务端保持路由和权限控制，以及一份构建产物在多环境部署的能力。
